# 에러 기반 디버깅 루프

## 에이전트와 디버깅할 때 가장 중요한 것

에이전트에게 "버그가 있어요"라고만 하면 도움을 받기 어렵습니다.  
**에러 메시지와 재현 방법을 정확하게 전달하는 것**이 핵심입니다.

---

## 디버깅 루프의 4단계

```
재현 → 에러 수집 → 에이전트에게 전달 → 수정 확인
```

### 1단계: 재현 (Reproduce)

버그를 안정적으로 다시 일으킬 수 있어야 합니다.  
"가끔 일어나는 버그"는 "항상 일어나는 버그"로 만들어야 에이전트가 도울 수 있습니다.

```
프롬프트 예시:
"python task_manager.py 를 실행하고 빈 문자열로 태스크를 추가하면
에러가 납니다. 직접 재현해서 확인해줘."
```

### 2단계: 에러 수집 (Collect)

에러가 나면 **터미널 출력 전체**를 복사합니다.  
스택트레이스(Traceback)에는 문제의 위치와 원인이 담겨 있습니다.

```
예시 스택트레이스:
Traceback (most recent call last):
  File "task_manager.py", line 45, in main
    task_id = tasks[-1]["id"] + 1
IndexError: list index out of range
```

이 정보에서 우리는 알 수 있습니다:
- **파일**: `task_manager.py`
- **줄**: 45번째 줄
- **에러 종류**: `IndexError` (빈 리스트에 접근)
- **원인 추정**: `tasks`가 비어있을 때 `tasks[-1]`에 접근

### 3단계: 에이전트에게 전달 (Report)

에러 메시지를 그대로 붙여넣고, 어떤 상황에서 발생했는지 설명합니다.

```
좋은 디버깅 프롬프트 구조:
1. "무엇을 했더니" (재현 방법)
2. "이런 에러가 났습니다" (스택트레이스 전체)
3. "예상했던 동작은" (올바른 동작 설명)
```

**예시:**
```
task_manager.py에서 태스크 목록이 비어있을 때
새 태스크를 추가하면 다음 에러가 납니다:

Traceback (most recent call last):
  File "task_manager.py", line 45, in add_task
    task_id = tasks[-1]["id"] + 1
IndexError: list index out of range

태스크가 없을 때도 id=1로 정상적으로 추가되어야 합니다.
이 버그의 원인을 찾아서 수정해줘.
```

### 4단계: 수정 확인 (Verify)

에이전트가 수정한 후, 반드시 직접 확인합니다.

```
- 에러가 나던 상황을 다시 재현해서 에러가 없어졌는지 확인
- 기존에 잘 동작하던 기능이 여전히 잘 동작하는지 확인
- 테스트가 있다면 pytest 실행
```

## 자주 쓰는 에러 유형별 접근법

### IndexError / KeyError

리스트나 딕셔너리에 없는 위치/키에 접근할 때 발생합니다.

```
에러: list index out of range
접근법: "tasks 리스트가 비어있을 때 처리가 빠져있을 수 있습니다. 확인해줘."
```

### TypeError

잘못된 타입으로 연산할 때 발생합니다.

```
에러: unsupported operand type(s) for +: 'int' and 'str'
접근법: "숫자와 문자열을 더하려고 했습니다. 타입 변환이 필요한 곳을 찾아줘."
```

### AttributeError

존재하지 않는 속성이나 메서드에 접근할 때 발생합니다.

```
에러: 'NoneType' object has no attribute 'title'
접근법: "None 체크 없이 속성에 접근하고 있습니다. None이 반환되는 경우를 처리해줘."
```

### AssertionError (테스트 실패)

테스트가 실패할 때 발생합니다. pytest의 출력을 그대로 붙여넣으면 됩니다.

```
에러:
FAILED test_task_manager.py::test_search_tasks - AssertionError: assert [] != [...]
접근법: 실패한 테스트 이름과 assert 내용을 그대로 에이전트에게 전달합니다.
```

## 에이전트 활용 디버깅 팁

### 원인 파악과 수정을 분리하기

```
나쁜 예: "이 버그 고쳐줘"
좋은 예: 
  1단계: "이 에러의 원인이 뭔지 설명해줘" → 이해
  2단계: "그 원인을 수정해줘" → 실행
```

원인을 이해하고 수정을 확인하면 같은 실수를 반복하지 않습니다.

### 수정 범위 제한하기

에이전트가 관련 없는 코드까지 바꾸지 않도록 범위를 명시합니다.

```
"add_task 함수 안에서만 수정해줘. 다른 함수는 건드리지 말아줘."
```

### 수정 전 코드 설명 요청

큰 변경 전에는 어떻게 고칠 것인지 먼저 설명을 듣습니다.

```
"수정하기 전에 어떻게 고칠 건지 먼저 설명해줘. 그다음에 수정할게."
```

## 핵심 정리

```
✅ 에러 메시지 전체를 붙여넣기 (일부만 넣으면 안 됨)
✅ 재현 방법을 구체적으로 설명
✅ 원인 파악 → 수정 순서로 진행
✅ 수정 후 반드시 재현해서 확인
```
