# TDD 스타일 에이전트 협업

## TDD가 에이전트와 잘 맞는 이유

TDD(Test-Driven Development)는 **테스트를 먼저 작성하고 구현하는** 개발 방식입니다.  
에이전트와 함께 쓸 때 특히 효과적인 이유는 두 가지입니다.

1. **요구사항이 명확해진다** — 테스트 코드가 "이 기능이 어떻게 동작해야 하는가"를 문서화합니다.
2. **검증이 자동화된다** — 에이전트가 코드를 수정할 때마다 테스트를 돌려 정상 동작을 즉시 확인할 수 있습니다.

```
요구사항 → 테스트 작성 → 구현 요청 → 테스트 실행 → 리뷰
```

---

## 기본 패턴: Red → Green → Refactor

### 1단계: Red (실패하는 테스트 작성)

아직 구현하지 않은 기능의 테스트를 먼저 작성합니다.  
에이전트에게 테스트 작성을 맡길 수도 있습니다.

```
프롬프트 예시:
"search_tasks(keyword) 함수를 만들 예정입니다.
이 함수는 tasks 리스트에서 title에 keyword가 포함된 항목만 반환합니다.
대소문자를 구분하지 않아야 합니다.
먼저 이 요구사항에 맞는 pytest 테스트를 작성해주세요. 구현은 하지 말고요."
```

### 2단계: Green (테스트를 통과하는 구현)

실패하는 테스트가 생겼으면, 그 테스트를 통과하는 가장 간단한 구현을 요청합니다.

```
프롬프트 예시:
"방금 작성한 test_search_tasks 테스트를 통과하는 search_tasks 함수를 task_manager.py에 구현해주세요."
```

### 3단계: Refactor (정리)

동작하는 코드가 생겼으면, 코드 품질을 개선합니다.

```
프롬프트 예시:
"search_tasks 구현이 동작하는 것을 확인했습니다.
코드를 더 읽기 좋게 리팩토링해주세요. 기능은 그대로 유지해야 합니다."
```

---

## 점진적 구현 원칙

한 번에 큰 기능을 요청하지 말고, **작게 쪼개서** 단계별로 요청합니다.

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "태스크 관리 시스템 전체를 만들어줘" | "태스크를 추가하는 add_task 함수만 먼저 만들어줘" |
| "검색·필터·정렬 기능을 추가해줘" | "먼저 키워드 검색만, 그다음 날짜 필터를 추가하자" |
| "버그를 다 고쳐줘" | "이 에러 메시지가 나오는 원인만 먼저 찾아줘" |

작게 나누면:
- 에이전트의 실수를 즉시 발견할 수 있다
- 어디서 문제가 생겼는지 쉽게 파악할 수 있다
- 중간에 방향을 바꾸기 쉽다

---

## 에이전트에게 테스트를 실행시키는 법

구현을 요청할 때 테스트 실행을 함께 요청하면, 에이전트가 동작을 스스로 검증합니다.

```
"search_tasks를 구현하고, 구현 후 pytest를 실행해서 테스트가 통과하는지 확인해줘."
```

에이전트는 다음 순서로 동작합니다:
1. 코드 작성
2. `pytest` 실행
3. 실패하면 스스로 코드 수정
4. 모든 테스트 통과 후 보고

이 사이클을 **"에이전트 루프"** 라고 합니다. 인간 개발자가 할 반복 작업을 에이전트가 대신합니다.

---

## 기존 테스트를 깨지 않기

새 기능을 추가할 때는 기존 테스트가 모두 통과하는지 항상 확인해야 합니다.  
에이전트에게 이를 명시적으로 지시하는 것이 좋습니다.

```
"search_tasks 기능을 추가할 때, 기존 테스트가 모두 통과하는지 확인하면서 진행해줘.
혹시 기존 테스트가 깨지면 즉시 알려줘."
```

---

## 핵심 정리

```
✅ 테스트를 먼저, 구현은 나중
✅ 한 번에 하나씩 작게
✅ 테스트 실행을 에이전트에게 위임
✅ 기존 테스트가 깨지면 즉시 수정
```
