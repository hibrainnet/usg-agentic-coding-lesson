# Claude Code 깊이 들여다보기

---

## 1. Claude Code란 무엇인가

**Claude Code**는 Anthropic이 만든 **터미널 기반 에이전트 코딩 도구(CLI)**입니다. 터미널에서 `claude`를 실행하면, AI 모델인 Claude가 여러분의 프로젝트 디렉토리 안에서 **직접 파일을 읽고, 수정하고, 명령을 실행하며** 작업을 수행합니다.

```
$ cd my-project          ← 작업할 프로젝트 폴더로 이동
$ claude                 ← 에이전트 실행
> 이 프로젝트 구조를 설명해줘
> 로그인 버그를 찾아서 고쳐줘
> 방금 고친 부분에 테스트를 추가하고 실행해줘
```

여기서 핵심은 `>` 뒤에 적은 것이 **명령어가 아니라 자연어 요청**이라는 점입니다. 여러분은 동료에게 말하듯 일을 부탁하고, Claude는 그것을 작업으로 바꿔 수행합니다.

---

## 2. "챗봇"이 아니라 "에이전트"인 이유

앞 문서의 핵심을 다시 짚습니다. 챗봇과 Claude Code의 결정적 차이는 **실제 행동 여부**입니다.

### 일반 챗봇 (예: 웹 채팅)
```
사람: 이 코드 버그 고쳐줘 (코드를 복사해서 붙여넣음)
챗봇: 12번 줄을 이렇게 바꾸세요 ...
사람: (답을 복사해서 → 에디터로 가서 → 붙여넣고 → 저장하고 → 직접 실행)
```
→ 챗봇은 **말만** 합니다. 실제 작업은 전부 사람이 손으로 합니다.

### Claude Code (에이전트)
```
사람: 이 코드 버그 고쳐줘
Claude: (직접 파일을 검색 → 읽기 → 수정 → 권한 요청 → 테스트 실행 → 결과 보고)
사람: (결과를 검토하고 승인)
```
→ Claude Code는 **직접 합니다.** 사람은 방향 제시와 검토에 집중합니다.

> 🔑 이 차이가 1교시 전체의 요점입니다. "말해주는 도구"에서 **"해주는 도구"**로의 전환.

## 3. 어디서 사용할 수 있는가

Claude Code는 여러 환경에서 동일한 엔진으로 동작합니다.

| 환경 | 설명 | 비고 |
|------|------|------|
| **터미널 (CLI)** | `claude` 명령으로 실행 | **이 강의의 기본 환경** |
| IDE 확장 | VS Code, JetBrains 안에서 사용 | 에디터와 통합 |
| 데스크톱 앱 | Mac / Windows 독립 앱 | GUI 선호 시 |
| 웹 | claude.ai/code | 설치 없이 |

> 💡 왜 터미널로 배우나? 터미널은 **어떤 환경에서도 통하는 가장 기본 형태**입니다. 여기서 익힌 개념은 IDE 확장이든 웹이든 그대로 적용됩니다. "가장 날것의 형태"를 먼저 익히면 나머지는 쉽습니다.

## 4. Claude Code의 핵심 특징

### (1) 프로젝트 전체를 다룬다
한 파일이 아니라 **코드베이스 전반**을 탐색하고 수정합니다. "이 기능과 관련된 파일을 다 찾아줘" 같은 요청이 가능합니다.

### (2) 도구를 직접 사용한다
Claude Code가 실제로 쓰는 도구들:
- **파일 읽기 / 쓰기** — 코드를 보고 고침
- **검색(grep/glob)** — 키워드·패턴으로 코드 탐색
- **셸 명령 실행** — 테스트 실행, 빌드, Git 등
- **Git 연동** — 커밋, 브랜치, PR (4교시)

### (3) 대화형으로 협업한다
계획을 먼저 보여주고, 사용자의 피드백을 반영하며 진행합니다. 한 번에 끝나지 않으면 "이렇게 해볼까요?"라고 묻고 조정합니다.

### (4) 컨텍스트를 기억하고 주입할 수 있다
`CLAUDE.md`라는 파일로 **프로젝트 규칙·관례**를 알려줄 수 있습니다. "우리 팀은 이런 코딩 스타일을 쓴다", "이 명령어로 테스트한다" 등을 한 번 적어두면 매번 설명할 필요가 없습니다. (3교시 집중)

### (5) 확장 가능하다
- **커스텀 슬래시 커맨드** — 자주 쓰는 작업을 나만의 명령으로
- **훅(Hooks)** — 특정 시점에 자동 동작 실행
- **MCP** — 외부 도구·데이터 소스 연결
- **서브에이전트** — 작업을 여러 에이전트에 분담
(모두 5교시에서 실습)

## 5. 내부 동작 원리: 도구(Tools) 시스템

Claude Code가 파일을 "마법처럼" 고치는 것처럼 보이지만, 실제로는 단순한 구조입니다.

```
LLM (Claude 모델) + 도구 묶음 = Claude Code
```

LLM은 여러분의 요청을 읽고 "어떤 도구를 어떤 순서로 쓸지" 결정합니다. 도구는 실제로 파일을 읽고 쓰고 명령을 실행하는 코드 조각입니다.

### Claude Code가 가진 도구 목록

| 분류 | 도구 | 하는 일 |
|------|------|---------|
| **읽기** | `Read` | 파일 내용 읽기 |
| **쓰기** | `Write` | 파일 생성 또는 전체 교체 |
| **수정** | `Edit` | 파일의 특정 부분만 수정 |
| **탐색** | `Glob` | 패턴으로 파일 찾기 (`**/*.py` 등) |
| **검색** | `Grep` | 파일 내용에서 문자열/패턴 검색 |
| **실행** | `Bash` | 셸 명령 실행 (pytest, git, npm 등) |
| **작업 관리** | `TodoWrite` | 복잡한 작업의 단계별 진행 상황 추적 |

### 요청 하나가 처리되는 실제 흐름

"로그인 버그 찾아서 고쳐줘"라고 입력하면 내부적으로 이런 일이 벌어집니다.

```
① [LLM 판단] "인증 관련 파일을 먼저 찾아야겠다"
      ↓
② [Glob 도구 호출] pattern="**/auth*.py"
      → 결과: ["auth.py", "tests/test_auth.py"]
      ↓
③ [LLM 판단] "auth.py를 읽어보자"
      ↓
④ [Read 도구 호출] path="auth.py"
      → 결과: (파일 전체 내용)
      ↓
⑤ [LLM 판단] "12번째 줄의 비교 로직이 잘못됐다. Edit 도구로 고치자"
      ↓
⑥ [사람에게 권한 요청] "이렇게 수정하겠습니다. 허용하시겠습니까?"
      ↓
⑦ [Edit 도구 호출] (수정 적용)
      ↓
⑧ [LLM 판단] "테스트를 돌려서 검증하자"
      ↓
⑨ [Bash 도구 호출] command="pytest test_auth.py -v"
      → 결과: 3/3 통과
      ↓
⑩ [LLM] "버그를 수정했습니다. 원인은 ..."
```

사람 눈에는 "Claude가 알아서 고쳤다"처럼 보이지만, 실제로는 **LLM이 여러 도구를 순서대로 선택·호출한 결과**입니다. 이 구조를 이해하면:

- 에이전트가 왜 어떤 파일을 먼저 보는지 이해할 수 있고
- 어떤 요청이 가능하고 어떤 것이 어려운지 예측할 수 있고
- 문제가 생겼을 때 어느 단계에서 막혔는지 파악할 수 있습니다.

## 6. 컨텍스트 윈도우: 에이전트의 "작업 기억"

LLM은 모든 것을 대화 기록 속에서 처리합니다. 이 대화 기록이 담길 수 있는 크기를 **컨텍스트 윈도우**라고 합니다.

### 컨텍스트 윈도우에 들어가는 것들

```
┌────────────────────────────────────────────────┐
│  컨텍스트 윈도우                                │
│                                                │
│  ① CLAUDE.md 내용 (프로젝트 규칙)              │
│  ② 지금까지의 대화 내용 (요청 + 응답)           │
│  ③ Read로 읽은 파일 내용                        │
│  ④ Bash 실행 결과 (테스트 출력 등)              │
│  ⑤ Glob/Grep 검색 결과                         │
└────────────────────────────────────────────────┘
```

### 컨텍스트가 중요한 이유

에이전트는 컨텍스트 윈도우 **밖의 내용은 모릅니다.** 세션을 시작하면 이전 대화를 기억하지 않는 이유가 바로 이것입니다.

| 상황 | 결과 |
|------|------|
| 짧은 대화, 작은 파일 | 컨텍스트 여유 있음 → 빠르고 정확 |
| 긴 대화, 큰 파일 여러 개 읽음 | 컨텍스트 가득 참 → 응답 느려지거나 초기 맥락 잊음 |
| 새 세션 시작 | 이전 대화 없음 → 처음부터 다시 |

### 실전 관리 방법

```
새 작업 시작 시  →  /clear      (대화 초기화, CLAUDE.md는 유지)
대화가 길어졌을 때  →  /compact  (대화를 요약해서 압축)
세션 재개 시  →  claude --continue  (이전 대화 불러오기)
```

> 🔑 규칙: **작업이 바뀌면 `/clear`** 합니다. 이전 작업의 맥락이 새 작업에 혼동을 주기 때문입니다.

## 7. 다른 도구와의 비교

| 도구 | 형태 | 작업 범위 | 실제 파일 접근 | 특징 |
|------|------|-----------|:---:|------|
| **Claude Code** | 터미널 에이전트 | 프로젝트 전체 | ✅ | 도구 사용·명령 실행·확장성 |
| GitHub Copilot | 에디터 인라인 | 현재 파일 위주 | △ | 빠른 자동완성·인라인 채팅 |
| Cursor | AI 통합 에디터 | 프로젝트 | ✅ | 에디터 자체에 AI 내장 |
| 웹 챗봇 (ChatGPT 등) | 웹 채팅 | 붙여넣은 코드만 | ❌ | 범용 대화, 코드 직접 접근 불가 |

> ⚠️ 다시 강조: 이 표는 우열을 가리는 게 아닙니다. **"실제 파일과 터미널에 접근해 작업을 수행하느냐"**가 Claude Code의 정체성이라는 점을 보여주는 것입니다.

## 8. 안전장치: 권한(Permission) 시스템

에이전트가 파일을 마음대로 고치고 명령을 실행한다면 불안하겠죠? Claude Code는 **위험할 수 있는 작업 전에 사용자에게 권한을 묻습니다.** 실제로 파일을 수정하기 직전에는 이런 **변경 미리보기**가 뜹니다:

```diff
  auth.py 파일을 수정하려고 합니다:

      stored_password = USERS[username]

-     if stored_password:
+     if password == stored_password:
          return True
      return False

  이 변경을 적용할까요?
  ❯ 1. 예 (허용)
    2. 예, 그리고 앞으로 이 파일 수정은 묻지 않기
    3. 아니오 (거부하고 다른 방법 제안)
```

명령을 실행하기 전에도 마찬가지로 묻습니다:

```text
  다음 명령을 실행하려고 합니다:
    $ pytest test_auth.py -v

  실행할까요?  [예 / 아니오]
```

- 파일 수정 / 명령 실행 시 **사용자 승인을 요청**합니다.
- 무엇을 하려는지 **미리 보여줍니다** (변경 내용 미리보기).
- 진행 중인 작업은 **ESC로 언제든 중단**할 수 있습니다.

> 🔑 좋은 습관: 처음에는 **기본(승인) 모드**로 쓰면서, 에이전트가 *무엇을 왜* 하려는지 매번 확인하세요. 익숙해지면 신뢰 범위를 넓혀갈 수 있습니다. (권한 모드는 2교시에서 자세히 다룸)

## 9. 작동 방식 한눈에 보기

앞 문서에서 배운 에이전트 루프가 Claude Code에서 실제로 어떻게 도느냐:

```
[사람]  "로그인 버그 고쳐줘"
   │
   ▼
[Claude] 탐색: auth 관련 파일 검색 → login.py 읽기
   │
   ▼
[Claude] 계획: "비밀번호 비교 로직이 잘못됨. 수정 후 테스트하겠음"
   │
   ▼
[Claude] 실행: login.py 수정 요청 → [사람이 권한 승인] → 수정
   │
   ▼
[Claude] 검증: pytest 실행 → 결과 확인
   │
   ▼
[사람]  결과 검토 → 만족하면 완료, 아니면 추가 요청
```

이 흐름을 1교시 실습에서 아주 작은 형태(`hello.txt` 만들기)로 직접 경험하게 됩니다.
