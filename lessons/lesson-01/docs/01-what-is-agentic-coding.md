# Agentic Coding이란 무엇인가?

---

## 0. 들어가며: 한 장면에서 시작하기

여러분이 회사에 인턴으로 들어간 첫날을 상상해 봅시다. 사수가 이렇게 말합니다.

> "우리 서비스에 로그인할 때 가끔 비밀번호가 틀려도 통과되는 버그가 있어. 한번 찾아서 고쳐줄래?"

이 한 문장을 받은 여러분은 무엇을 할까요?

1. 먼저 **코드베이스를 둘러봅니다.** 로그인 관련 파일이 어디 있는지 찾습니다.
2. 관련 코드를 **읽고 이해합니다.** 인증 로직이 어떻게 흘러가는지 파악합니다.
3. 문제가 될 만한 부분을 **추론하고 가설을 세웁니다.**
4. 코드를 **고치고**, 테스트를 **돌려서** 정말 고쳐졌는지 **확인합니다.**
5. 안 되면 다시 1~4를 **반복합니다.**

여기서 핵심은, 사수는 *"무엇을(What)"* 만 말했지 *"어떻게(How)"* 는 말하지 않았다는 점입니다. "12번째 줄의 `==`를 `===`로 바꿔"라고 하지 않았습니다. 여러분이 알아서 탐색하고, 판단하고, 실행하고, 검증했습니다.

**Agentic Coding은 바로 이 "주니어 개발자"의 역할을 AI 에이전트가 맡는 방식입니다.** 그리고 여러분은 이제 인턴이 아니라, 그 인턴에게 일을 맡기고 결과를 검토하는 **사수**의 자리로 올라갑니다.

---

## 1. 정의: Agentic Coding의 세 가지 핵심

> **Agentic Coding이란, AI 에이전트가 스스로 코드를 탐색·계획·수정·검증하는 과정을 사람이 감독하고 협업하며 소프트웨어를 만들어가는 개발 방식이다.**

이 정의를 세 조각으로 뜯어봅시다.

### (1) "에이전트(Agent)" — 행동하는 AI

일반 챗봇은 **말로 답합니다.** "이렇게 하세요"라고 알려줄 뿐, 실제로 무언가를 하지는 않습니다.
반면 **에이전트**는 **행동합니다.** 목표가 주어지면, 그 목표를 달성하기 위해 스스로 도구(파일 읽기, 코드 수정, 명령 실행 등)를 사용합니다.

### (2) "스스로 탐색·계획·수정·검증" — 작업의 자율성

에이전트는 단일 동작이 아니라 **여러 단계를 스스로 이어서** 수행합니다. 파일을 찾고, 읽고, 고치고, 테스트를 돌리고, 실패하면 다시 시도합니다. 이 자율적 반복이 "자동완성"과 결정적으로 다른 지점입니다.

### (3) "사람이 감독하고 협업" — 인간이 운전석에

자율적이라고 해서 사람이 빠지는 것이 아닙니다. 사람은 여전히 **목표를 정하고, 방향을 제시하고, 결과를 검증**합니다. 좋은 Agentic Coding은 "AI에게 다 맡기는 것"이 아니라 **"AI와 함께 일하는 것"**입니다.

> 🔑 기억할 비유: **자율주행과 운전자.**
> 차가 스스로 운전하더라도, 운전자는 목적지를 정하고, 위험할 때 개입하고, 최종 책임을 집니다. Agentic Coding에서 여러분은 운전자입니다.

## 2. 코딩 도구는 어떻게 진화해 왔는가

Agentic Coding이 왜 "지금" 등장했는지 이해하려면, 개발 도구의 진화를 짚어야 합니다.

```
[1세대] 문법 강조 · 기본 자동완성        ── 글자/토큰 단위
        (예: 옛날 IDE의 변수명 자동완성)
           │
           ▼
[2세대] AI 코드 제안                      ── 줄/블록 단위
        (예: GitHub Copilot)
        "다음에 올 코드를 예측해서 제안"
           │
           ▼
[3세대] Agentic Coding                    ── 작업(Task) 단위
        (예: Claude Code)
        "작업 전체를 위임받아 수행"
```

### 1세대: 사전 같은 도구

옛날 IDE의 자동완성은 "내가 입력한 변수가 무엇인지" 정도를 알려줬습니다. 도움은 됐지만, 모든 판단과 작성은 사람의 몫이었습니다. **똑똑한 사전**에 가깝습니다.

### 2세대: 다음 줄을 예측하는 동료

GitHub Copilot 같은 도구는 한 단계 나아갔습니다. 여러분이 함수 이름을 쓰면, 그 안의 코드를 **통째로 제안**합니다. 하지만 본질은 여전히 **"다음에 올 코드 예측"**입니다.

- 한 번에 한 파일, 보통 커서 주변만 봅니다.
- 제안을 받아들일지 말지는 **사람이 매 줄 결정**합니다.
- 코드를 *작성*하는 것을 돕지만, 일을 *대신 수행*하지는 않습니다.

### 3세대: 일을 위임받는 동료

Agentic Coding은 단위가 다릅니다. "이 함수의 다음 줄"이 아니라 **"이 버그를 고쳐"** 또는 **"이 기능을 추가해"** 같은 **작업 전체**를 위임받습니다.

- 한 파일이 아니라 **프로젝트 전체를 탐색**합니다.
- 코드를 읽고, 여러 파일을 수정하고, **명령을 실행**하고, 결과를 봅니다.
- 사람은 매 줄이 아니라 **작업 단위로 검토**합니다.

## 3. 비교로 명확히 하기: 자동완성 vs Agentic Coding

| 구분 | 코드 자동완성 (예: Copilot) | Agentic Coding (예: Claude Code) |
|------|------------------------------|-----------------------------------|
| **작업 단위** | 한 줄 / 블록 | 하나의 작업 (기능 추가, 버그 수정 등) |
| **주도권** | 사람이 모든 줄을 직접 작성 | 에이전트가 여러 파일을 탐색·수정 |
| **도구 사용** | 없음 (텍스트 제안만) | 파일 읽기/쓰기, 명령 실행, 검색, Git |
| **맥락 범위** | 현재 파일 주변 | 프로젝트 전체 + 지정한 규칙(CLAUDE.md) |
| **상호작용** | 단방향 제안 | 대화형 (계획 → 실행 → 피드백 루프) |
| **검증** | 사람이 수동으로 | 에이전트가 테스트/실행으로 자가 검증 가능 |
| **비유** | 똑똑한 사전 / 빠른 타이피스트 | 함께 일하는 주니어 동료 |

> ⚠️ 오해 주의: 이 표는 "Copilot이 나쁘다"는 뜻이 아닙니다. 둘은 **쓰임이 다른 도구**입니다. 빠른 타이핑 보조가 필요하면 자동완성이, 작업 위임이 필요하면 에이전트가 적합합니다. 실제로 둘을 함께 쓰기도 합니다.

## 4. 핵심 개념: 에이전트의 작업 루프 (Agent Loop)

Agentic Coding의 심장은 아래 **반복 루프**입니다. 앞으로 6시간 동안 하는 거의 모든 실습이 이 루프의 변형입니다. 외워두면 좋습니다.

```
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  ▼                                                  │
탐색(Explore) ─→ 계획(Plan) ─→ 실행(Act) ─→ 검증(Verify) ─┘
  │                │             │             │
관련 파일 찾기      접근법 수립    코드 수정      테스트/실행
코드 읽기          순서 정하기    명령 실행      결과 확인
구조 파악          위험 점검                     (실패 시 루프 반복)
```

### 단계별로 보기

**① 탐색 (Explore)**
에이전트가 가장 먼저 하는 일은 "어디를 봐야 하는가"입니다. 키워드로 코드를 검색하고, 관련 파일을 열어 읽습니다.
- 예: "로그인 로직이 어디 있지?" → `auth.py`, `login.py` 검색 → 파일 읽기

**② 계획 (Plan)**
무엇을, 어떤 순서로 바꿀지 설계합니다. Claude Code에는 실행 전에 계획을 먼저 보여주는 **계획 모드(Plan Mode)**가 있어, 사람이 검토 후 진행할 수 있습니다 (3교시에서 실습).
- 예: "비밀번호 비교 연산자가 잘못됐다. 이 부분을 고치고, 관련 테스트를 추가하겠다."

**③ 실행 (Act)**
실제로 파일을 수정하거나 명령을 실행합니다. 위험할 수 있는 동작 전에는 **사용자에게 권한을 묻습니다.**
- 예: `login.py` 수정, `pytest` 실행

**④ 검증 (Verify)**
정말 의도대로 됐는지 확인합니다. 테스트를 돌리거나 앱을 실행해 결과를 봅니다. **실패하면 다시 탐색/계획으로 돌아갑니다.**
- 예: 테스트 실행 → 통과하면 완료, 실패하면 원인 분석 후 루프 반복

> 🔑 검증의 중요성: 루프에서 가장 자주 생략되지만 가장 중요한 단계가 **검증**입니다. "코드가 생성됐다 ≠ 코드가 맞다." 이 강의 내내 강조할 핵심입니다.

## 4-1. 실제로 보기: 루프가 도는 장면

먼저, 버그가 있는 작은 로그인 코드가 있다고 합시다. (`examples/demo-login/auth.py`)

```python
def authenticate(username: str, password: str) -> bool:
    if username not in USERS:
        return False
    stored_password = USERS[username]

    # 🐞 비밀번호를 '비교'하지 않고, 존재 여부만 확인한다 → 틀려도 통과!
    if stored_password:
        return True
    return False
```

이 버그 때문에, 비밀번호가 틀려도 로그인이 통과합니다. 직접 실행해 보면:

```bash
$ python3 auth.py
환영합니다, alice님!
환영합니다, alice님!      ← 'WRONG' 비밀번호인데 통과해버림 (버그)
로그인 실패: 사용자명 또는 비밀번호가 올바르지 않습니다.
```

이제 Claude Code에게 **작업을 통째로 위임**합니다. 아래는 실제 세션의 흐름입니다.

```text
$ claude

> test_auth.py를 실행하면 test_wrong_password가 실패해.
  원인을 찾아서 고치고, 다시 테스트를 돌려서 통과하는지 확인해줘.

● 먼저 테스트를 실행해 현재 상태를 확인하겠습니다.          ← [탐색]
  $ pytest test_auth.py -v
    test_correct_password PASSED
    test_wrong_password   FAILED
    test_unknown_user     PASSED

● auth.py를 읽어 원인을 찾겠습니다.                          ← [탐색]
  (auth.py 읽음)

● 원인을 찾았습니다. authenticate()가 비밀번호를 비교하지 않고  ← [계획]
  단지 저장된 값이 존재하는지만 확인합니다. 비교하도록 고치겠습니다.

  다음과 같이 수정하려고 합니다:                              ← [실행: 권한 요청]
    auth.py
    - if stored_password:
    + if password == stored_password:
  [허용하시겠습니까?]  → (사용자가 '허용' 선택)

● 수정했습니다. 테스트를 다시 실행합니다.                     ← [검증]
  $ pytest test_auth.py -v
    test_correct_password PASSED
    test_wrong_password   PASSED   ← 이제 통과!
    test_unknown_user     PASSED

● 버그를 수정했고 모든 테스트가 통과합니다.
  원인은 비밀번호 비교 누락이었습니다.
```

이 한 장면 안에 **탐색 → 계획 → 실행(+권한) → 검증** 루프가 모두 들어 있습니다.

> 🔑 마지막 단계를 다시 보세요. Claude는 고친 뒤 **스스로 테스트를 돌려 검증**했습니다. 하지만 그 결과가 정말 맞는지 최종 확인하는 건 여전히 사람의 몫입니다 — "위임하되, 검증하라."

## 5. 구체적 시나리오: 같은 작업, 세 가지 방식

"사용자 등록 시 이메일 형식이 올바르지 않으면 에러를 내는 기능을 추가하라"는 작업을 세 방식으로 비교해 봅시다. 목표 코드는 대략 이런 모습입니다:

```python
def register(email: str) -> str:
    if "@" not in email:                       # ← 추가하려는 검증
        raise ValueError("올바르지 않은 이메일 형식입니다")
    # ... 기존 등록 로직 ...
    return f"{email} 등록 완료"
```

### 방식 A: 손으로 직접 (도구 없음)
1. `register` 함수가 어느 파일에 있는지 **직접 검색해 연다**
2. 검증 로직을 **직접 타이핑한다**
3. 테스트 코드를 **직접 작성하고 실행한다**
```bash
$ grep -rn "def register" .      # 직접 찾고
$ vim users.py                   # 직접 열어 고치고
$ pytest                         # 직접 검증
```
→ 탐색·작성·검증 **모든 단계가 사람의 시간**

### 방식 B: 자동완성 (2세대)
1. `register` 함수를 **직접 찾아 연다**
2. `if "@" not in` 까지 치면 나머지를 **제안받아** 빠르게 완성
```python
    if "@" not in email:        # ← 여기까지 타이핑하면
        raise ValueError(...)   # ← 이 줄이 자동 제안됨
```
3. 테스트는 **직접 작성·실행**
→ **작성(타이핑)만 빨라짐.** 탐색과 검증은 여전히 사람 몫

### 방식 C: Agentic Coding (3세대)
1. 자연어로 **요청만** 한다:
```text
> register 함수에 이메일 형식 검증을 추가해줘.
  '@'가 없으면 ValueError를 던지고, 관련 테스트도 추가해서 돌려줘.
```
2. 에이전트가 `register`를 **찾아 읽고 → 검증 로직 추가 → 테스트 작성 → 실행**까지 수행
3. 사람은 **변경 내용을 검토**하고 승인
→ 사람은 **방향 제시 + 검토**에 집중. 탐색·작성·검증을 위임

### 한눈에 비교

| 단계 | A: 손으로 | B: 자동완성 | C: Agentic |
|------|:---:|:---:|:---:|
| 파일 찾기(탐색) | 사람 | 사람 | **에이전트** |
| 코드 작성 | 사람 | 사람+제안 | **에이전트** |
| 테스트 작성·실행(검증) | 사람 | 사람 | **에이전트** |
| 최종 판단·검토 | 사람 | 사람 | **사람** |

## 6. 왜 하필 지금 중요한가

### (1) 모델의 능력이 임계점을 넘었다
LLM이 긴 맥락을 이해하고, 도구를 신뢰성 있게 사용하고, 여러 단계를 스스로 이어갈 수 있게 되면서 비로소 "작업 위임"이 현실이 됐습니다.

### (2) 생산성의 차원이 바뀐다
반복적이고 탐색적인 작업(낯선 코드 분석, 보일러플레이트 작성, 테스트 생성, 리팩터링)을 위임하면, 개발자는 **설계와 본질적 문제 해결**에 집중할 수 있습니다.

### (3) 진입 장벽이 낮아진다
낯선 언어, 낯선 프레임워크, 거대한 레거시 코드베이스에 적응하는 시간이 극적으로 줄어듭니다. "이 코드가 뭐 하는 거야?"를 즉시 물어볼 수 있는 동료가 항상 옆에 있는 셈입니다.

### (4) 요구되는 역량이 재편된다
> "코드를 직접 타이핑하는 능력"만큼이나
> **"에이전트에게 잘 위임하고, 그 결과를 비판적으로 검증하는 능력"**이 중요해집니다.

이것이 여러분이 4학년 시점에 이 강의를 듣는 이유입니다. 곧 현업에서 만날 작업 방식이기 때문입니다.

---

## 7. 한계와 위험: 반드시 알아야 할 것

Agentic Coding을 **맹신하면 위험합니다.** 강의 내내 균형 잡힌 시각을 유지하기 위해, 한계를 분명히 짚고 갑니다.

### (1) 할루시네이션 (그럴듯한 오답)
에이전트는 때때로 **존재하지 않는 함수나 잘못된 API를 자신 있게** 만들어냅니다. "자신 있어 보인다"와 "맞다"는 다릅니다.

### (2) 맥락의 한계
에이전트는 여러분 머릿속의 비즈니스 요구사항, 회사 관례, 말하지 않은 제약을 알지 못합니다. **알려주지 않으면 모릅니다.** (그래서 3교시에서 `CLAUDE.md`를 배웁니다.)

### (3) 검증 책임은 사람에게
에이전트가 만든 코드의 **최종 책임은 사람**입니다. 테스트하지 않은 코드를 그대로 배포하는 것은, 검토하지 않은 인턴의 코드를 그대로 운영에 올리는 것과 같습니다.

### (4) 보안·민감 정보
민감한 코드나 비밀 키를 다룰 때는 주의가 필요합니다. 무엇을 에이전트에게 맡길지 판단하는 것도 개발자의 역량입니다.

> 🔑 이 강의의 원칙: **"위임하되, 검증하라 (Delegate, but Verify)."**
> 6시간 내내 이 문장으로 돌아올 것입니다.

## 8. 흔한 오해 바로잡기

| 오해 | 진실 |
|------|------|
| "AI가 다 해주니 개발자는 필요 없어진다" | 검증·설계·판단은 여전히 사람 몫. 오히려 **기초 실력이 더 중요**해진다. 잘못된 코드를 알아보려면 실력이 있어야 한다. |
| "프롬프트만 잘 쓰면 된다" | 프롬프트는 입구일 뿐. **결과를 읽고 검증하는 능력**이 진짜 실력. |
| "에이전트 결과는 믿어도 된다" | 에이전트도 틀린다. **항상 검토하고 테스트**해야 한다. |
| "초보자에게 더 유리하다" | 부분적으로만 맞다. 초보자는 *틀린 결과를 알아채지 못해* 더 위험할 수 있다. |
| "Agentic Coding은 그냥 챗봇이다" | 챗봇은 말하고, 에이전트는 **행동(파일 수정·명령 실행)**한다. |

---

## 9. 이 강의에서 기르는 4가지 능력

1. **명확한 위임** — 에이전트가 오해 없이 일하도록 작업을 정의하는 법 (3교시 집중)
2. **비판적 검증** — 에이전트의 결과를 읽고, 테스트하고, 신뢰할지 판단하는 법 (4교시 집중)
3. **자동화·확장** — 반복 작업을 커맨드/훅/MCP로 확장하는 법 (5교시 집중)
4. **안전한 협업 습관** — 효율적이고 안전하게 에이전트와 일하는 워크플로우 (6교시 종합)
