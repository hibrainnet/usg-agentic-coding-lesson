# 베스트 프랙티스와 주의점

## 에이전트 출력을 신뢰하는 법

에이전트는 강력하지만 실수를 합니다. 올바른 신뢰 수준을 유지하는 것이 중요합니다.

### 항상 검증하라

```
❌ 에이전트가 "완성됐습니다"라고 했으니 됐겠지
✅ 에이전트가 완성했다고 해도, 직접 실행해서 확인한다
```

검증 체크리스트:
- 코드를 수정했으면 → 테스트 실행
- 기능을 추가했으면 → 직접 실행해서 동작 확인
- 버그를 고쳤으면 → 버그가 나던 상황을 다시 재현

### 에이전트 오류 패턴

| 오류 유형 | 예시 | 예방법 |
|-----------|------|--------|
| 관련 없는 코드 수정 | "이 함수만 고쳐줘"에서 다른 함수도 변경 | 범위를 명시: "이 함수 안에서만 수정해줘" |
| 과도한 구현 | 간단한 기능 요청에 복잡한 클래스 생성 | "가장 간단한 방법으로" 명시 |
| 존재하지 않는 API 사용 | 실제로 없는 라이브러리 함수 사용 | 실행해서 ImportError/AttributeError 확인 |
| 테스트 통과 = 정상 동작 혼동 | 테스트가 잘못 작성되어 있어도 통과 | 테스트 로직도 직접 검토 |

## 효율적인 협업 워크플로우

### 1. 컨텍스트 유지 전략

대화가 길어질수록 에이전트의 응답 품질이 떨어질 수 있습니다.

```
- 긴 세션은 /compact로 주기적으로 압축
- 새로운 독립 작업은 새 세션에서 시작
- CLAUDE.md에 프로젝트 규칙을 명시해서 매번 설명 생략
```

### 2. 작업 단위 나누기

```
✅ 한 번의 요청 = 한 가지 명확한 작업
❌ 한 번의 요청 = 여러 기능 + 여러 버그 수정 + 리팩토링
```

작게 나누면:
- 실수를 즉시 발견
- 어디서 문제가 생겼는지 파악 쉬움
- 필요하면 `/rewind`로 되돌리기 쉬움

### 3. 불확실할 때 먼저 물어보기

큰 변경 전에는 에이전트의 계획을 먼저 확인합니다.

```
"이 기능을 추가하기 전에 어떻게 구현할 것인지 설명해줘.
코드는 나중에 작성해줘."
```

계획이 마음에 들면 "좋아, 진행해줘"로 실행합니다.

## 에이전트의 한계

### 에이전트가 잘 하는 것

```
✅ 반복적인 코드 생성 (CRUD, 보일러플레이트)
✅ 알려진 패턴의 버그 수정 (IndexError, None 체크 등)
✅ 코드 탐색 및 요약
✅ 테스트 작성
✅ 문서화
```

### 에이전트가 잘 못하는 것

```
⚠️ 완전히 새로운 알고리즘 설계
⚠️ 시스템 전체 아키텍처 결정
⚠️ 비즈니스 요구사항 해석
⚠️ 최신 프레임워크 (학습 데이터에 없는 버전)
⚠️ 미묘한 성능 최적화
```

**사람의 판단이 필요한 영역과 에이전트가 효율적인 영역을 구분하는 것이 핵심 역량입니다.**

## 보안과 안전

### 신뢰하지 말아야 할 것

```
❌ 에이전트가 생성한 코드를 검토 없이 프로덕션에 배포
❌ 에이전트에게 민감한 정보(API 키, 비밀번호) 직접 입력
❌ 에이전트가 제안한 패키지를 검토 없이 설치
```

### 안전한 사용

```
✅ 변경 사항은 항상 git diff로 검토 후 커밋
✅ API 키는 환경 변수로 분리, CLAUDE.md에는 키 이름만 언급
✅ 새 패키지는 npm/pypi 페이지에서 실제 존재 여부 확인
```

## 나만의 워크플로우 정립

수업을 통해 배운 도구를 조합해서 자신의 워크플로우를 만들어봅니다.

```
예시 워크플로우:
1. CLAUDE.md에 프로젝트 규칙 작성
2. 기능 요청 → 계획 모드로 먼저 설계
3. 테스트 먼저 → 구현 순서 (TDD)
4. /review 커맨드로 변경 사항 확인
5. /commit 커맨드로 메시지 생성
6. 직접 실행해서 최종 검증
```

이 워크플로우는 고정이 아닙니다. 실습하면서 자신에게 맞는 방식으로 조정합니다.
