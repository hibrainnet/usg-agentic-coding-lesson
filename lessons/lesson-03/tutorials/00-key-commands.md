# [실습] 핵심 슬래시 커맨드 실전 활용

> 🎯 목표: 2교시에서 설치하고 처음 써본 Claude Code의 커맨드를 **언제·왜 쓰는지** 실제 시나리오로 익힌다.
> ⏱️ 예상 시간: 15분
> 💡 `examples/contact-book/` 폴더에서 진행합니다.

---

## 들어가며

2교시에서 `/help`, `/clear`, `/exit` 세 가지를 배웠습니다. 이번에는 개발 흐름 안에서 자연스럽게 쓰게 될 커맨드를 시나리오별로 익힙니다. **커맨드를 외우는 것이 목적이 아닙니다** — 어떤 상황에서 무엇을 쓰면 좋은지 감을 잡는 것이 목적입니다.

```bash
cd 경로/lessons/lesson-03/examples/contact-book
claude
```

---

## 시나리오 1: 작업 전 현황 파악 (2분)

새 세션을 시작하면 가장 먼저 현재 상태를 확인합니다.

```
> /status
```
→ 어떤 계정과 모델로 연결됐는지 확인합니다.

```
> /usage
```
→ 이번 세션에서 쓴 토큰과 비용을 확인합니다. 처음엔 거의 0입니다.

```
> /context
```
→ 컨텍스트 윈도우가 얼마나 남았는지 시각적으로 확인합니다.

**언제 쓰나**: 긴 작업 세션 중에 `/usage`와 `/context`를 주기적으로 확인하면 비용과 성능 저하를 예방할 수 있습니다.

---

## 시나리오 2: 대화가 길어졌을 때 (3분)

contact-book에 대해 여러 질문을 해보겠습니다.

```
> contacts.py의 전체 구조를 설명해줘.
> storage.py는 어떻게 동작해?
> test_contacts.py에서 테스트가 몇 개야?
> 각 함수의 타입 힌트 현황을 알려줘.
```

대화가 쌓였습니다. 지금 `/context`를 확인하면 이전보다 차 있을 겁니다.

같은 작업을 계속 이어가야 한다면:
```
> /compact
```
→ 대화를 요약·압축합니다. 이전 대화의 핵심은 기억하면서 토큰을 아낍니다.

**`/clear`와의 차이 다시 확인**:
- `/compact` — 지금 하는 작업을 계속 이어갈 때 (맥락 유지)
- `/clear` — 작업이 끝나고 새 작업을 시작할 때 (맥락 초기화)

이 실습이 끝나면 `/clear`로 초기화하고 다음 실습으로 넘어갑니다.
```
> /clear
```

## 시나리오 3: 속도가 너무 느리거나, 더 깊이 분석이 필요할 때 (3분)

작업 성격에 따라 모델을 조절할 수 있습니다.

```
> /model
```
→ 현재 모델과 선택 가능한 모델 목록이 표시됩니다.

빠른 단순 작업이 많다면 가벼운 모델로, 복잡한 설계 문제라면 강력한 모델로 전환합니다.

```
> /effort
```
→ 추론 깊이를 조절하는 슬라이더가 표시됩니다. `low`는 빠르고 저렴하며, `high`·`max`는 더 깊이 생각합니다.

**실습**: `/effort low`로 설정하고 간단한 질문을 해봅니다. 그다음 `/effort high`로 바꾸고 같은 질문의 응답 품질 차이를 관찰합니다.

```
> /effort low
> add_contact 함수에서 개선할 점이 있으면 알려줘.
```
```
> /effort high
> add_contact 함수에서 개선할 점이 있으면 알려줘.
```

---

## 시나리오 4: 코드 품질 빠르게 확인 (4분)

코드를 수정한 뒤 빠르게 리뷰하고 싶을 때:

```
> /code-review low
```
→ 명확한 버그와 오류만 빠르게 찾습니다.

```
> /code-review high
```
→ 버그뿐 아니라 설계·성능·가독성까지 검토합니다.

```
> /simplify
```
→ 버그가 아닌 코드 정리(중복 제거, 가독성 향상)만 제안합니다.

**실습**: 먼저 `/code-review low`를 실행하고 결과를 확인한 뒤, `/code-review high`를 실행해서 얼마나 더 많은 내용이 나오는지 비교합니다.

---

## 시나리오 5: 실수했을 때 되돌리기 (3분)

에이전트가 잘못된 방향으로 코드를 수정했을 때:

**방법 1 — ESC**: 작업 중에 ESC를 누르면 즉시 중단됩니다.

**방법 2 — `/rewind`**: 마지막 응답을 취소하고 이전 상태로 돌아갑니다.
```
> /rewind
```

**방법 3 — `/branch`**: 현재 지점에서 새 분기를 만들어 실험합니다. 실험이 실패해도 원본은 유지됩니다.
```
> /branch 이메일검증-실험
```
→ 새 접근법을 시도하고, 마음에 들지 않으면 이전 분기로 돌아옵니다.

**실습**: 간단한 요청을 한 뒤 `/rewind`를 실행해서 대화가 되돌아가는 것을 확인합니다.
```
> add_contact에 print 디버그 로그를 추가해줘.
> /rewind
```

## 정리: 상황별 커맨드 선택

| 상황 | 커맨드 |
|------|--------|
| 세션 시작 전 상태 확인 | `/status`, `/usage`, `/context` |
| 같은 작업 중 대화가 길어짐 | `/compact` |
| 새 작업으로 전환 | `/clear` |
| 응답이 너무 느림 | `/model` (가벼운 모델로), `/effort low` |
| 복잡한 문제, 정확도 필요 | `/effort high` 또는 `/model` (강력한 모델로) |
| 빠른 코드 리뷰 | `/code-review low` |
| 심층 코드 리뷰 | `/code-review high` 또는 `ultra` |
| 코드 정리만 | `/simplify` |
| 마지막 응답 취소 | `/rewind` |
| 다른 접근법 실험 | `/branch` |
