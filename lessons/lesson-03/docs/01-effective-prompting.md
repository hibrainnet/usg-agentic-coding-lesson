# 효과적인 프롬프팅

## 왜 프롬프트가 중요한가

Claude Code는 여러분이 말한 것을 그대로 실행합니다. "말한 것"이 모호하면 에이전트도 모호하게 행동하고, "말한 것"이 명확하면 에이전트도 명확하게 행동합니다.

같은 결과를 원하더라도, 어떻게 요청하느냐에 따라 결과의 품질이 크게 달라집니다.

```
"코드 개선해줘"              → 무엇을? 어떻게? 범위는?
"이 함수에 입력 검증 추가해줘" → 조금 더 나음, 하지만 아직 부족
"add_contact()에 이메일 형식 검증을 추가해줘.
 '@'와 '.'가 없으면 ValueError 발생.
 validate_email()로 분리하고 테스트 2건 작성."
                            → 에이전트가 오해할 여지가 없음
```

## 나쁜 프롬프트의 3가지 패턴

### 패턴 1: 너무 막연하다

```text
❌ "이거 고쳐줘"
❌ "코드 개선해줘"
❌ "더 좋게 만들어줘"
```

에이전트는 "더 좋다"의 기준을 모릅니다. 성능? 가독성? 타입 안전성? 에러 처리? 아무것도 지정하지 않으면 에이전트가 임의로 선택합니다.

### 패턴 2: 범위가 없다

```text
❌ "이 프로젝트 리팩터링해줘"
```

"이 프로젝트" 전체를 수정하면 예상치 못한 파일까지 건드릴 수 있습니다. 어느 파일의 어느 함수인지 지정하세요.

### 패턴 3: 맥락이 없다

```text
❌ "에러 처리 추가해줘"
```

어떤 에러? 어떤 방식으로? 에러 메시지는 한국어? 영어? 예외를 던져야 하나, 출력만 해야 하나?

---

## 좋은 프롬프트의 5가지 구성 요소

좋은 프롬프트에는 다음 다섯 가지가 담겨 있습니다. 전부 필요한 건 아니지만, 빠질수록 결과가 모호해집니다.

```
① 대상(Target)   — 어디를? 어느 파일/함수?
② 작업(Task)     — 무엇을? 추가/수정/리팩터링/테스트?
③ 조건(Condition)— 어떻게? 규칙, 형식, 언어, 제약
④ 범위(Scope)    — 어디까지? 건드릴 파일, 건드리지 말 파일
⑤ 검증(Verify)   — 어떻게 확인? 테스트 작성, 실행 결과 확인
```

### 비교 예시 1: 기능 추가

| 나쁜 프롬프트 | 좋은 프롬프트 |
|------|------|
| "이메일 검증 추가해줘" | "contacts.py의 add_contact()에 이메일 형식 검증을 추가해줘. '@'가 없으면 ValueError('올바르지 않은 이메일 형식입니다') 발생. 빈 문자열은 허용. validate_email() 함수로 분리. test_contacts.py에 테스트 2건 추가." |

### 비교 예시 2: 버그 수정

| 나쁜 프롬프트 | 좋은 프롬프트 |
|------|------|
| "버그 있어" | "contacts.py의 delete_contact()를 실행하면 ID가 없을 때 IndexError가 발생해. 에러 메시지를 '오류: ID {id}에 해당하는 연락처가 없습니다.'로 출력하고 정상 종료하도록 고쳐줘. contacts.py만 수정할 것." |

### 비교 예시 3: 리팩터링

| 나쁜 프롬프트 | 좋은 프롬프트 |
|------|------|
| "storage.py 리팩터링해줘" | "storage.py의 load_contacts()와 save_contacts()에 타입 힌트를 추가해줘. load는 `list[dict]`, save의 contacts 파라미터는 `list[dict]`로. JSON 파싱 실패 시 JSONDecodeError를 잡아 빈 목록 반환 + '경고: contacts.json 파싱 실패' 출력. storage.py만 수정할 것." |

## 실전 패턴 모음

### 패턴 A: 단계 지정

복잡한 작업은 단계를 명시하면 에이전트가 순서를 임의로 바꾸지 않습니다.

```text
> 아래 순서대로 진행해줘:
> 1. validate_email() 함수를 contacts.py에 추가
> 2. add_contact()에서 validate_email() 호출
> 3. test_contacts.py에 테스트 3건 추가
> 4. pytest 실행해서 통과 확인
```

### 패턴 B: 건드리지 말 것 명시

의도치 않은 수정을 방지합니다.

```text
> contacts.py만 수정할 것. storage.py와 test_contacts.py는 건드리지 마.
```

### 패턴 C: 먼저 설명, 그다음 실행

이해 없는 수정을 방지합니다.

```text
> 먼저 어떻게 수정할지 설명만 해줘. 내가 괜찮다고 하면 그때 실행해줘.
```

### 패턴 D: 예시 제공

원하는 결과물의 형태를 예시로 주면 오해가 줄어듭니다.

```text
> 출력 형식을 아래처럼 바꿔줘:
> 현재: [1] 홍길동 — 010-1234-5678
> 목표: [001] 홍길동 / 010-1234-5678 / hong@example.com
```
