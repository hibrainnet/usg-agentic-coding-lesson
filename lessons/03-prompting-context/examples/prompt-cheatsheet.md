# 프롬프트 작성 참고 카드

Claude Code에게 요청할 때 이 카드를 참고하세요.

---

## 좋은 프롬프트의 5요소

| 요소 | 질문 | 예시 |
|------|------|------|
| **① 대상** | 어느 파일/함수? | `contacts.py의 add_contact()` |
| **② 작업** | 무엇을? | `이메일 형식 검증 추가` |
| **③ 조건** | 어떻게? | `'@' 없으면 ValueError, 빈 문자열은 허용` |
| **④ 범위** | 어디까지? | `contacts.py만 수정, storage.py 건드리지 말 것` |
| **⑤ 검증** | 어떻게 확인? | `테스트 2건 추가 후 pytest 실행` |

---

## 요청 유형별 템플릿

### 기능 추가

```
> [파일명]의 [함수명]에 [기능]을 추가해줘.
>
> 조건:
> - [조건 1]
> - [조건 2]
>
> 범위: [파일명]만 수정할 것
> 검증: [테스트 방법]
```

### 버그 수정

```
> [함수명]에서 [증상]이 발생해.
> [재현 방법 또는 에러 메시지]
>
> 수정 조건:
> - [조건]
>
> [파일명]만 수정할 것. 수정 전에 원인을 먼저 설명해줘.
```

### 리팩터링

```
> [파일명]의 [함수/클래스]를 리팩터링해줘.
>
> 목표:
> - [목표 1 (예: 타입 힌트 추가)]
> - [목표 2 (예: 공통 로직 함수 분리)]
>
> 건드리지 말 것: [다른 파일명]
> 기존 동작은 바뀌지 않아야 함. 테스트로 검증해줘.
```

### 설명 요청

```
> [파일명]의 [함수명]이 하는 일을 설명해줘.
> 특히 [궁금한 부분]이 어떻게 동작하는지 알고 싶어.
```

### 테스트 작성

```
> [함수명]에 대한 테스트를 [테스트 파일명]에 추가해줘.
>
> 케이스:
> - 정상 케이스: [설명]
> - 예외 케이스: [설명]
>
> 테스트 후 pytest 실행해서 통과 확인해줘.
```

## 유용한 패턴

### 먼저 설명, 그다음 실행

```
> 먼저 어떻게 할지 설명만 해줘. 내가 승인하면 그때 실행해줘.
```

### 건드리지 말 것 명시

```
> [파일A]와 [파일B]만 수정할 것. 나머지는 건드리지 마.
```

### 단계 지정

```
> 아래 순서대로 진행해줘:
> 1. [단계 1]
> 2. [단계 2]
> 3. 테스트 실행
```

### 예시 제공

```
> 출력 형식을 아래처럼 바꿔줘:
> 현재: [현재 출력]
> 목표: [원하는 출력]
```

## 피해야 할 표현

| 나쁜 표현 | 문제점 | 대안 |
|----------|------|------|
| "이거 고쳐줘" | 대상 불명 | "contacts.py의 XX 함수를 고쳐줘" |
| "코드 개선해줘" | 기준 불명 | "타입 힌트와 docstring을 추가해줘" |
| "리팩터링해줘" | 범위 불명 | "storage.py만 리팩터링해줘" |
| "더 좋게 만들어줘" | 목표 불명 | "가독성을 위해 긴 함수를 분리해줘" |
| "전부 다 바꿔줘" | 범위 과다 | 작업을 쪼개서 파일 단위로 요청 |

## CLAUDE.md 체크리스트

프로젝트 루트의 CLAUDE.md에 다음이 있는지 확인하세요.

- [ ] 프로젝트 실행 명령어 (`python3 main.py ...`)
- [ ] 테스트 실행 명령어 (`pytest tests/ -v`)
- [ ] 사용 언어/스타일 규칙 (에러 메시지 언어, 타입 힌트 여부 등)
- [ ] 파일/모듈 역할 요약 (어디에 무엇이 있는지)
- [ ] 새 기능 추가 시 규칙 (테스트 필수 여부 등)
